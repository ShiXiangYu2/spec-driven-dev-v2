# 反 Spec Slop 清单

**Spec slop = AI 生成的"看起来对但实际上没用"的规范。**

## 什么是 Spec Slop？

Spec slop 是 AI 默认模式下产出的"规范最大公约数"——模糊的 What、抽象的 Constraints、敷衍的 Non-Goals。这些 spec 读起来像那么回事，但开发者拿到手后仍然不知道具体该做什么。

**为什么反 spec slop 很重要**：
1. 用户请你写 spec，是要**减少沟通成本和返工**
2. AI 默认产出的 spec = 训练语料的平均 = **没有任何项目特异性**
3. 所以 AI 默认 spec = 帮用户把项目稀释成"又一个 AI 写的文档"
4. 反 spec slop 不是文字洁癖，是**替用户保护规范的执行力**

## 核心症状与修复

### 症状 1：What 太模糊

**❌ Slop 示例**：
```markdown
## 做什么（What）
实现用户登录功能
```

**问题**：没有验收标准，开发者不知道"完成"的定义是什么。

**✅ 正确做法**：
```markdown
## 做什么（What）
用户可以用邮箱+密码登录：
- 邮箱格式验证（符合 RFC 5322）
- 密码长度 8-64 字符，必须包含字母+数字
- 登录成功返回 JWT token（有效期 7 天）
- 连续 5 次失败锁定账户 15 分钟
- 错误提示不暴露具体是邮箱错还是密码错
```

**修复公式**：What = 具体行为 + 可测量标准 + 边界条件

---

### 症状 2：Non-Goals 敷衍或缺失

**❌ Slop 示例**：
```markdown
## 非目标
- 暂时不做高级功能
- 后续版本再做
```

**问题**：AI 看了等于没看，仍然会去实现"高级功能"。

**✅ 正确做法**：
```markdown
## 非目标（v1 不做）
- OAuth 第三方登录（Google/GitHub/微信）
- 手机号 + 验证码登录
- "记住我"功能（持久化登录状态）
- 登录日志 / 审计功能
- 多设备登录管理（踢下线/设备列表）
```

**修复公式**：Non-Goals = 具体功能名 + 括号解释（防止歧义）

---

### 症状 3：Constraints 太抽象

**❌ Slop 示例**：
```markdown
## 约束
- 代码要高质量
- 性能要好
- 安全性要强
```

**问题**：无法执行，无法验证。

**✅ 正确做法**：
```markdown
## 约束
- 密码必须 bcrypt 加密，cost factor = 12
- JWT token 有效期 7 天，refresh token 30 天
- API 响应时间 P99 < 200ms
- 单元测试覆盖率 ≥ 80%
- 所有外部 API 调用必须有 5 秒超时 + 3 次重试
```

**修复公式**：Constraints = 具体技术 + 具体数值 + 可验证方式

---

### 症状 4：任务拆解太粗

**❌ Slop 示例**：
```markdown
## 任务拆解
- [ ] 实现登录功能
- [ ] 写测试
- [ ] 做前端页面
```

**问题**：每个任务都很大，无法估算时间，无法分配执行。

**✅ 正确做法**：
```markdown
## 任务拆解（2-5 分钟粒度）
- [ ] 创建 User 模型（email, password_hash, failed_attempts, locked_until）
- [ ] 实现 bcrypt 密码哈希工具函数（含 salt 生成）
- [ ] 实现邮箱格式验证中间件（RFC 5322）
- [ ] 实现 POST /api/v1/auth/login 路由处理器
- [ ] 实现登录失败计数和账户锁定逻辑
- [ ] 实现 JWT token 生成（7 天）和验证
- [ ] 写单元测试：密码正确时返回 200 + token
- [ ] 写单元测试：密码错误时返回 401
- [ ] 写单元测试：连续 5 次失败后返回 429
- [ ] 写集成测试：完整登录流程（注册→登录→访问受保护接口）
```

**修复公式**：任务 = 单一职责 + 可命名文件/函数 + 2-5 分钟完成

---

### 症状 5：缺少执行计划

**❌ Slop 示例**：
```markdown
Spec 写完了。你可以开始写代码了。
```

**问题**：开发者不知道从哪里开始，哪些文件要改。

**✅ 正确做法**：
```markdown
## 执行计划

### 文件变更清单
| 文件 | 操作 | 说明 |
|------|------|------|
| src/models/User.js | 新增 | User 模型定义 |
| src/utils/crypto.js | 新增 | bcrypt 封装 |
| src/routes/auth.js | 新增 | 登录路由 |
| src/middleware/validator.js | 新增 | 输入验证 |
| tests/auth.test.js | 新增 | 测试文件 |

### 执行顺序（按依赖）
1. 模型层 → 工具层 → 中间件 → 路由 → 测试
```

**修复公式**：执行计划 = 文件级清单 + 依赖顺序 + 验证点

---

### 症状 6：忽略现有代码库

**❌ Slop 示例**：
```markdown
## 技术约束
- 后端框架：Express.js
- ORM：Prisma
```

**问题**：代码库已经是 Fastify + TypeORM 了，spec 与之冲突。

**✅ 正确做法**：
先执行代码库资产协议：
```bash
# 读取现有配置
cat package.json  # 确认框架版本
cat src/config/database.js  # 确认 ORM
```

然后写 spec：
```markdown
## 技术约束（基于代码库现状）
- 后端框架：Fastify v4.x（与现有代码一致）
- ORM：TypeORM v0.3.x（与现有代码一致）
- 数据库：PostgreSQL 15（与现有配置一致）
- **新增依赖**：bcrypt（密码哈希）、jsonwebtoken（JWT）
```

**修复公式**：已有代码库 → 先读再写，约束从代码中提取

---

### 症状 7：接口设计只有路径

**❌ Slop 示例**：
```markdown
## API 设计
POST /login
```

**问题**：没有请求体格式、响应格式、错误码，前端无法对接。

**✅ 正确做法**：
```markdown
## API 设计

```
POST /api/v1/auth/login
```

**请求体**：
```json
{
  "email": "string, required, 邮箱格式",
  "password": "string, required, 8-64 字符"
}
```

**成功响应（200）**：
```json
{
  "success": true,
  "data": {
    "token": "string, JWT token",
    "expires_in": 604800
  }
}
```

**错误响应**：
- 400：参数格式错误
- 401：认证失败（邮箱或密码错误）
- 429：登录尝试过于频繁
- 500：服务器内部错误
```

**修复公式**：接口 = 路径 + 方法 + 请求体 + 响应体 + 错误码 + 状态码约定

## 快速自检清单

写完 spec 后，问自己这 7 个问题：

- [ ] What 是否能写出至少 3 个测试用例？
- [ ] Non-Goals 是否有具体功能名（不是"高级功能"这种抽象词）？
- [ ] Constraints 是否有至少 1 个具体数值？
- [ ] 任务拆解是否能估算出每个任务的完成时间？
- [ ] 是否有文件级变更清单？
- [ ] 如果是已有代码库，spec 是否引用了现有技术栈？
- [ ] 接口设计是否包含请求/响应/错误码？

7 条全满足 = 好 spec。任何一条不满足 = 需要修复。
